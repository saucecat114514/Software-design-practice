## 1 背景与问题陈述

需求基线 **BL-20260623-01** 已经 CCB 评审冻结：明确 88 条功能需求、5 条接口需求、17 条非功能需求，覆盖招生客户、教务、财务、经营 BI、系统管理五大子系统及一个综合横切域（见知识图谱 KG-EDU-V1：37 业务组件 / 6 子系统 / 25 约束 / 10 基础模块）。进入软件设计阶段，必须**先确定唯一顶层架构风格**，作为后续模块划分（MDS/DTS）、约束设计（TLCD）、AI 代码生成、逆向校验（RCR）的最高依据。

当前痛点与决策必要性：

1. **AI 自由生成会导致架构混杂**：若不预先固化分层规则与依赖方向，AI 会把训练样本里的坏习惯（如在 OrderService 里直接拼 SQL、在 Controller 里写业务逻辑、跨层穿透调用）写进代码，造成层次错乱、依赖无序。
2. **业务本质是事务型系统**：知识图谱显示业务以 CRUD + 审批流 + 对账勾稽 + 报表为主，非算法/计算密集，需要一种**职责清晰、可模式化复制、可自动化校验**的结构。
3. **量化约束（来自 SRS §3.3 与 CCB 裁决）**：
   - 并发/容量：约 100 并发 / 约 500 用户；关键路径（报名、缴费、查名额）响应 ≤3s，试听 P99≤3s、目标≤1s；
   - 可靠性：核心库灾备 RTO 30min、RPO≈5min，全字段不可篡改审计、财务数据留存≥3 年；
   - 安全：RBAC×校区双维隔离、高危操作告警、备份加密；
   - 可配置化：CCB 8 项量化约束（缓冲 5min[1~30]、折价系数 1.0[0~1]、发票阈值 5000、续费 7 天/3 次、请假率 20%/30% 等）要求规则参数化、参数快照冻结。
4. **时间窗口**：本决策是六步工作流第一步，是 ADR-002~004、TLCD、v1 代码的不可逆前置；不先定则全部下游无法启动。

## 2 约束前提（不可突破边界）

- **技术栈约束**：Java + Spring 技术栈（项目既定决策），与四层分层天然对齐。
- **业务约束**：必须支撑多校区数据隔离 + 集团汇总、规则可配置化、全链路审计留痕。
- **性能约束**：关键路径秒级响应（≤3s），但无秒杀级超高并发诉求。
- **团队约束**：1 人全量执行（AI 辅助），无多团队并行；架构认知负担必须低。
- **运维约束**：阿里云单体部署（实训本机等效），无独立 SRE，不具备复杂分布式运维条件。
- **工程约束**：架构规则必须可转化为 AI 生成提示词 + 可被 CodeGraph 自动化漂移校验（机器可读、可执行、可量化）。
- **合规约束**：遵循 IEEE 830 / GB/T 9385；架构决策须可双向溯源至 RTM 基线。

|---|---|---|---|---|---|---|---|
| **A 四层分层架构** | Controller→Service→Repository→Model 单向分层，单体部署 | 结构规范、职责清晰、AI 模式化生成一致性最高、可自动化校验、落地与运维成本最低 | 分层链路略长，极致性能优化空间有限 | 中小型事务型 CRUD/审批/对账系统、单人或小团队、单体部署 | 低 | **4.82** | 不淘汰（选定） |

## 4 最终决策

系统全局采用 **【Controller → Service → Repository → Model 四层分层架构】**，单工程单体部署，强制单向依赖。落地标准：

- **Controller 层**：仅做参数校验、请求路由、响应封装、鉴权入口；不写业务逻辑、不直接访问数据。
- **Service 层**：承载全部业务逻辑、事务边界、审批/对账/算价等业务规则；可调用工具类、缓存、消息、其它 Service。
- **Repository 层**：唯一数据访问出口，封装对数据库的 CRUD；不含业务逻辑。
- **Model 层**：实体/DO/DTO/VO 等数据载体，无行为逻辑。
- **依赖方向**：Controller → Service → Repository → Model，**严格单向，禁止反向与跨层穿透**。
- **辅助手段（非主风格）**：缓存、消息通知、定时任务、审批流引擎等以**基础模块/工具**形态被 Service 调用，不构成第二主架构风格；少数富规则模块（退费计算、收入确认、对账勾稽）可在 Service 内用充血领域服务局部吸收，**不引入整体 DDD**。
- **适用范围**：本决策适用于全系统 5 个 Spring 服务（crm-service / academic-service / finance-service / bi-service / admin-service）及 platform-common 基础设施，无例外模块。

## 5 决策理由

1. **匹配业务本质**：知识图谱证实业务以事务型 CRUD + 审批 + 对账为主（88 FR 跨 37 组件），分层的"表现-业务-数据"三段切分恰好对齐这种事务流，职责边界天然清晰。
2. **AI 代码生成适配性最高（课程核心）**：分层是高度模式化结构，AI 按固定层级与单向依赖生成代码的一致性最高，且最易被 CodeGraph 正/逆向图谱比对校验；这是「AI 赋能软件工程」实训能否闭环的关键。
3. **单人 + 实训的硬约束**：分层认知负担最低、零额外基础设施、单体部署运维最简，落地成本最低——DDD 的建模纪律与微服务的分布式运维对单人都不可持续。
4. **量化诉求被满足而非过度**：100 并发/500 用户用分层 + 缓存 + 索引即可达成秒级响应；多校区用数据隔离 + 配置满足，无需微服务级弹性；可配置化约束在 Service 层 + 配置模块即可承载。
5. **可演化留口**：若未来某子系统并发或团队规模跃升，可在保持分层的前提下对该域做模块化拆分或局部服务化（走 ADR-005 变更闭环），不必现在为不存在的规模付费。

## 6 后果与风险评估

### 6.1 正向收益
- 架构统一规范、代码层级一致性高，跨模块认知成本低；
- AI 生成代码可预测、可校验，RCR 架构维度漂移风险极低、通过率高；
- 落地快、运维简单，把精力集中到业务正确性与变更闭环（评分核心）上。

### 6.2 负面代价
- 分层链路固定，极致单点性能优化空间有限（可用缓存、索引、读写分离弥补）；
- 单体在超大规模下存在伸缩上限（当前量级无此问题）。

### 6.3 潜在风险
- **人为/AI 跨层调用**：Controller 直接调 Repository、Service 拼 SQL、下层反向依赖上层等架构违规；
- **业务逻辑外溢**：把业务规则写进 Controller 或 Repository，导致分层名存实亡；
- **单体膨胀**：随功能增长单工程变臃肿，模块边界模糊。

### 6.4 风险应对预案
- 由 **ASD-EDU-V1 + TLCD 的 C-ARCH 约束**固化"四层单向、禁穿透"规则，写入 AI 生成提示词；
- 由 **CodeGraph 逆向校验（D15 RCR）**自动检测跨层/反向/穿透依赖，漂移分级整改（Blocker 必修）；
- 由 **ADR-002~004** 进一步收敛模块粒度、职责与数据边界、依赖白名单，防单体膨胀。

## 7 AI 适配约束规则（机器可识别，可写入提示词 / CodeGraph 校验）

1. 严格遵循 **Controller → Service → Repository → Model 单向依赖**；上层可调下层，**下层禁止反向调用上层**。
2. **禁止跨层穿透**：Controller 禁止直接调用 Repository 或操作数据库；禁止任意层跨过相邻层调用非相邻层。
3. **业务逻辑收敛 Service**：Controller 不含业务分支判断；Repository 不含业务逻辑；Service 不拼裸 SQL（统一经 Repository/ORM）。
4. **数据访问收敛 Repository**：所有 DB 操作仅出现在 Repository 层。
5. **分层目录强制**：代码按 `controller / service / repository / model`（及 `config / common`）分目录存放，目录即层级，AI 生成必须归位。
6. **无第二主风格**：缓存/消息/定时/审批等以工具或基础模块被 Service 调用，AI 不得据此演化出事件驱动或微服务结构。
7. **可配置化优先**：CCB 8 项量化阈值不得硬编码，须经配置模块注入（对应 NFR/CCB 约束）。

## 9 验收与校验标准（可量化 / 可自动化）
1. 系统全量代码严格按 Controller/Service/Repository/Model 四层目录划分，无错位文件；
2. RCR 架构维度校验：**无架构风格漂移、无层级穿透、无反向依赖**（漂移 Blocker = 0）；
3. 全量代码不存在 Controller 直接访问数据库、Service 拼裸 SQL、下层反向引用上层的情况；
4. CCB 量化阈值 100% 经配置注入，代码中无硬编码魔数；
5. 期末 RCR 总漂移 ≤ 2 处（红线）。

## 10 迭代与作废条件
本基线为**初始化架构基准**，归档后禁止直接修改。仅当出现以下情形，方可经 **ADR-005 及以上编号**发起架构重构变更：
- 单子系统并发或数据量跃升到单体无法支撑（需局部服务化）；
- 团队规模扩张到多团队并行（需服务边界隔离）；
- 引入与分层根本冲突的技术范式（如全面事件驱动）。
任何变更必须走 CR → CIA → ADR-005 → 新基线闭环，旧决策标「已替代」并归档，不释放编号。
