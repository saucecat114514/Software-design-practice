## DEF-001 退费/审批"二级中间档"为 SRS 推断
- **定位**：`FR-REF-003 审批退费`、`FR-ORDER-003 优惠分级审批` 的业务规则。
- **缺陷**：CCB ISSUE-008 只原生裁决两档——「≤5000元且≥9折→校区主管，30分钟超时自动通过」与「>10000元或<85折或>150课时→校长，绝不自动通过」。对 **5000~10000元 / 85折~90折** 的中间区间，CCB 未原生裁决；A4 生成、A5 复核时按"就高不就低"补出"二级通道（4h/8h 超时自动通过）"。
- **根因**：补访/裁决阶段未穷举折扣-金额二维分档，留下规则空洞，由下游推断填补。
- **影响**：中额退费/折扣的风控严格度取决于 SRS 推断，未经业务方与 CCB 正式确认，存在风控口径风险。
- **v2 变更建议（CR 候选①）**：发起 CR 就中间档的审批级别与是否允许超时自动通过征询校长/财务，CCB 正式裁决后修订 FR-REF-003 / FR-ORDER-003，并经 CIA 评估对审批引擎与对账报警的影响。

## DEF-002 A3 缺关键流程时序图
- **定位**：`wiki/summaries/uml/`（现有 6 用例图 + 6 活动图，无时序图）。
- **缺陷**：SRS规范 §5.3 与 RTM规范 §5.3 均把"关键流程时序图"列为基线建模产物，但 A3 仅产出用例图与活动图；缺跨服务/含外部系统的时序图，如：①家长小程序缴费→支付渠道回调→收款入账→报名→课消确认；②退费审批→发票红冲→财务系统凭证推送；③试听一键锁定的并发独占时序。
- **根因**：D06 建模聚焦"用例 + Guard 显式化活动图"，时序图未排入首版。
- **影响**：接口交互的时间顺序、异步回调与超时边界仅靠 SRS/IFR 文字表达，设计阶段（MDS/DTS、契约 OAS）缺时序输入。
- **v2 变更建议（CR 候选②）**：补 2~3 张核心 PlantUML 时序图，纳入基线建模产物并回填 RTM「建模ID」列。

## DEF-003 SRS NFR 未分配条目级 ID
- **定位**：`SRS §3.3 非功能需求`（性能/可靠/安全/可维护/可扩展/易用/合规 7 子节）。
- **缺陷**：NFR 以子节成段描述，未给每条指标分配 `NFR-{类型}-NNN`。A6 建 RTM 时只能从需求清单 NFR 型 REQ 反推、补出 17 条 NFR 映射；SRS 正文内仍无条目 ID，正向「SRS 段落 → NFR-ID」不严格一一对应。
- **影响**：NFR 的测试用例绑定与变更影响定位粒度偏粗（按子节而非按条）。
- **v2 变更建议（CR 候选③）**：SRS 升版时为每条 NFR 指标编号并回填 RTM，实现 NFR 条目级双向追溯。

## DEF-004 转班单价口径与退费单价口径未对齐
- **定位**：`FR-TRF-002 转班费用计算`（用"实际支付单价"）vs `FR-REF-001/TREF-001 退费`（用"含赠送总课时均价"，CCB ISSUE-007）。
- **缺陷**：转班"剩余价值"= 剩余正课 × 实际支付单价（不含赠课摊薄）；退费单价 = 实缴总额 ÷（购买+赠送）总课时（赠课摊薄）。两套口径并存；CCB ISSUE-011 只定了转班"退旧买新"模型，未明确转班剩余价值是否同走 ISSUE-007 口径（A5 VF-006教务 已标，A6 未细化，属备案）。
- **影响**：同一学员"先转班再退费"或"含赠课转班"场景下，剩余价值因口径不同可能产生差额争议。
- **v2 变更建议（CR 候选④）**：CR 征询财务，统一或明确区分两场景单价口径，修订 FR-TRF-002 与 FR-REF/TREF；经 CIA 评估对转班/退费引擎与利润表的影响。

## DEF-005 通知渠道降级与回执核对缺可测 SLA
- **定位**：`IFR-MSG-001 统一通知接口`，及 `FR-SCH-004 / FR-RSCH-003 / FR-ATT-007` 等多处"推送通知 + 追踪已读"。
- **缺陷**：渠道优先级（微信 > 短信 > 站内信，ISSUE-004）已定，但**渠道失败的降级触发条件、重试次数与回执核对时限**缺量化 SLA（如"微信下发失败 ≤N 秒降级短信""回执未达 ≤M 分钟生成待办"）。
- **影响**：家校通知触达率不可度量；调课/对账通知的"已读闭环"存在盲区，与 FR 中"次日生成未读待办"难以精确联动。
- **v2 变更建议（CR 候选⑤）**：补通知中心的降级 SLA 与回执核对量化指标，纳入 IFR-MSG-001 与相关 FR 的验收标准。

## DEF-006 备案默认值待真实业务方正式复核
- **定位**：`CCB_BL-20260623-01_评审记录.md §二`（8 项备案：缓冲 5min[1~30]、折价系数 1.0[0~1]、发票阈值 5000、续费 7/3/3、请假率 周20%月30%、灾备 RTO 30min、转介绍来源保留、撞单姓名脱敏）。
- **缺陷**：这些默认值多为合理推断或单方 REQ，标注"可配置/待确认"，尚未经真实业务方（财务/校长）在正式评审中签字确认。
- **影响**：上线/试运行时若实际阈值与默认不符，需走配置或需求变更。
- **v2 变更建议（CR 候选⑥）**：试运行期收集真实阈值，凡偏离默认即发起 CR 调整配置项或需求，闭环更新基线。

## DEF-007 A4 对长结构无防截断后处理
- **定位**：`run_srs.py gen()`（未复用 `run_modeling.py extract_puml` 的"取末块"逻辑）。
- **缺陷**：A4 数据章 E-R 曾因 `chat_complete` 截断后模型重画，导致 mermaid 画两遍、围栏奇数（已在 v0.1 手工修）。A4 运行器仍缺"检测截断重画 → 取最后一个完整块"的自动后处理。
- **影响**：工作流1 自动跑 A4 时，数据/架构等含长结构的章节仍可能产出重复内容或坏围栏，削弱"一键全自动"的可靠性，依赖人工兜底。
- **v2 变更建议（工程改进⑦）**：把"取最后一个 \`\`\`...\`\`\` / @startuml..@enduml 完整块"下沉为 `llm_config` 公共函数，run_srs / run_verification / run_modeling 统一复用。

## 小结
- 共识别 **7 条缺陷**（重要 4 / 建议 3），覆盖 需求口径、建模覆盖、接口健壮、工程可靠 四个层面，均**有据可查、定位到条**。
- 其中 **6 条**（DEF-001~006）适合作为第 3-4 周「需求变更闭环」的 **CR 埋点**；DEF-007 为工程健壮性改进，可在工作流1 增强时一并落实。
- 这些缺陷不在初始基线内修改；将择 1~2 条（优先 DEF-001 中间档审批 或 DEF-004 单价口径，业务价值与变更影响均显著）走完整 **CR→CIA→建模/SRS 更新→基线比对(DIFF)→CCB→新基线 BL-…-02→ADR-005** 闭环，作为期末核心评分项。
