## 一、补访澄清结果（按涉众）

### 课程顾问 补访澄清

- **ISSUE-001**
  - **原疑点**：优惠规则“模糊/互斥”缺乏可量化的判定标准，无法落地为系统自动判断逻辑。
  - **涉众明确答复**：
    - **两套逻辑分流**：（1）制度里明确禁止的组合（如教材费不参与折扣），系统应直接拦截并提示违反的具体规则编号；（2）制度未明确允许/禁止、存在解释空间的组合，系统应按最大优惠方案试算并自动触发主管审批。
    - **附带规则**：拦截与试算审批的响应时间应在2秒内完成。
  - **需求修订建议**：
    1. 新增“优惠互斥规则表”后台维护功能，字段包含：互斥优惠项A、互斥优惠项B、禁止叠加、所依据的制度编号。
    2. 订单选择优惠时增加实时校验：命中互斥规则则弹窗拦截，文案模板：“根据【优惠制度第X条】，该组合不可用，请重新选择”，禁止提交。
    3. 未命中互斥但涉及跨类或标记为“模糊区域”的组合：实时计算最大优惠金额并展示，订单生成时携带“叠加明细+[特殊叠加待审批]”标签。
    4. 拦截弹窗与待审批订单生成的整体响应时间须≤2秒。
  - **新状态**：`已澄清`

- **ISSUE-003**
  - **原疑点**：试听名额“数据实时”验收标准缺失，未定义数据一致性时延上限。
  - **涉众明确答复**：
    - **延迟阈值**：理想状态≤1秒，可接受极限为3秒，超过5秒将导致前台“炸锅”和客户投诉。
    - **并发冲突处理**：发生名额抢占时，系统必须实时校验并立即返回“名额已被占用”，同时提供“加入候补”入口，严禁出现虚假成功。
  - **需求修订建议**：
    1. 明确跨终端数据同步延迟指标：核心要求为P99≤3秒，目标≤1秒。
    2. 并发测试验收标准：A端锁定最后名额后，B端在1.5秒内发起查询或锁定同一名额，必须返回明确阻断信息（名额已满/已占用）。
    3. 锁定失败界面必须包含“加入候补”单次点击入口。
    4. 将超过5秒的同步延迟定义为系统故障，触发实时告警。
  - **新状态**：`已澄清`

- **ISSUE-005**
  - **原疑点**：候补优先级排序存在“付费类型队列”与“手动意向优先级”两套未调和的逻辑，系统无法执行。
  - **涉众明确答复**：
    - **主次排序规则明确**：第一关键字（刚性）为【是否付费试听】，先分高级队列（已付费）和普通队列（未付费）；第二关键字（柔性）为【顾问手动标注的紧急度】，在高级队列内部排序。
    - **排序细则**：紧急度分“1天内”、“3天内”、“1周内”等。高级队内按紧急度降序，相同紧急度按进入候补时间升序。普通队列默认按进入候补时间，允许顾问手动加意向标签微调。
  - **需求修订建议**：
    1. 候补列表UI分为“高级队列（已付费试听）”与“普通队列（未付费）”两个分区展示。
    2. 新增候补记录“紧急度”字段（枚举值：1天内、3天内、1周内），由顾问手动选择。
    3. 高级队列排序规则：紧急度降序 > 进入候补时间升序。
    4. 普通队列排序规则：默认进入候补时间升序；若顾问添加意向标签，则以标签权重 > 进入时间排序。
    5. 释放名额后的通知推送，必须遵循“高级队列全员 > 普通队列全员”的推送顺序。
  - **新状态**：`已澄清`

- **ISSUE-008**
  - **原疑点**：课程顾问期望2分钟内快速审批，与校长风控诉求可能延长审批时间相悖，存在冲突。
  - **涉众明确答复**：仍未明确
  - **需求修订建议**：N/A
  - **新状态**：`待CCB裁决`（课程顾问立场：通过分级审批机制解决，小额/常规折扣授权校区主管并设5分钟超时自动通过；大额/深度折扣须校长审批，但需设值班校长或代理人机制及15分钟倒计时提醒，以平衡销售效率与风控。此建议需校长方确认。）

- **ISSUE-022**
  - **原疑点**：试听锁定15分钟倒计时是否需叠加教务的转换缓冲时间进行冲突检测，可能导致锁定成功但实际无法衔接。
  - **涉众明确答复**：仍未明确
  - **需求修订建议**：N/A
  - **新状态**：`待CCB裁决`（课程顾问立场：试听名额锁定必须基于含缓冲时段的真实可用时间进行冲突检测，确保锁定的时段资源有效，避免到场后无法衔接。此规则需与教务方协调确认。）

- **ISSUE-023**
  - **原疑点**：课程顾问要求统一缴费渠道杜绝个人收款码，与财务要求支持线下收款拆分功能相冲突。
  - **涉众明确答复**：仍未明确
  - **需求修订建议**：N/A
  - **新状态**：`待CCB裁决`（课程顾问立场：线下收款拆分功能必须限定由财务人员操作，且全程在系统内记录，严禁顾问经手资金或使用个人码。此安全底线需与财务方共同确认流程。）

### 财务人员 补访澄清

**ISSUE-007**
- **原疑点**：退费基础单价存在财务（实缴总额÷购买+赠送）与教务（实缴总额÷购买，赠课不折现）两种互斥算法。
- **涉众明确答复**：财务坚持采用 **实缴总额 ÷ (购买课时 + 赠送课时)** 作为退费单价，此为财务红线；家长端建议增加 **“赠课剩余价值 = ¥0.00”** 展示并附计算说明（如“退费单价 = 实缴总额 / 总课时（含赠课）”），以对齐预期。
- **需求修订建议**：
  1. 在退费计算引擎中硬编码此单价算法，退费单生成时自动计算且不可手动修改；同一订单多次退费保持单价恒定。
  2. 家长端退费页展示 `<退费单价 = 实缴总额 / 总课时（含赠课）>`，并独立显示一行 `赠课剩余价值：¥0.00`，备注“赠课价值已折算至每节课单价中”。
- **新状态**：`待CCB裁决`（财务立场：强制采用含赠课的总课时均摊单价，家长端补充分布信息；教务口径冲突待裁决）

**ISSUE-019**
- **原疑点**：退费规则变更后，对已购未退订单适用购买时规则还是申请时规则不明确。
- **涉众明确答复**：必须按 **学员购买成交时的规则版本** 执行；系统须在订单支付成功时强制保存完整的 **退费规则快照**（含退费单价公式、各阶段退费比例、手续费标准等），退费时只读快照、不查当前规则表。
- **需求修订建议**：
  1. 订单支付成功后自动将当时生效的退费规则元数据写入订单快照字段 `refund_rule_snapshot`，该快照不可手动修改。
  2. 退费计算引擎仅读取订单快照；后台修改退费规则不影响已有订单快照。
  3. 增加规则变更日志与按版本追溯影响订单的功能（变更时间、变更人、变更对比、覆盖订单列表）。
- **新状态**：`已澄清`

**ISSUE-020**
- **原疑点**：财务的“待认领”与系统管理员的“单边账报表”可能重复或定义不同。
- **涉众明确答复**：统一命名为 **“单边待认领账”**；由 **财务主导处理**，入口设在 **财务收费台账的支付流水列表**；财务手动核实（依学员姓名、金额、时间等）并关联到订单；认领成功后金额 **直接并入正常收入**，月末对账报表单独列示 **“本月单边账认领金额”** 明细行；学员端不提供自助认领。
- **需求修订建议**：
  1. 支付流水列表将“支付成功但无对应订单”的流水统一标识为“单边待认领账”，提供筛选与高亮。
  2. 仅财务角色可发起认领操作，关联已有订单或补单后关联；认领后流水状态变为“已认领”，金额计入关联订单的实缴总额，并入正常收入，无特殊标记；所有认领操作留痕（操作人、时间、流水号、学员ID）。
  3. 月末对账报表增加“本月单边账认领金额”明细行，该行不计入网关流水总额（防重复），作为对账调节项。
  4. 系统管理后台提供单边账报表，可标注认领状态并导出。
- **新状态**：`已澄清`

**ISSUE-023**
- **原疑点**：课程顾问要求统一缴费渠道、杜绝个人收款码，与财务需支持线下收款拆分冲突。
- **涉众明确答复**：接受“**线下拆分仅限财务操作且必须系统留痕**”、“顾问/教务完全不碰现金或线下款”的边界；补充：线下拆分仅限学员已通过官方渠道全额支付后的特殊场景，绝不可常态化；拆分操作需强制记录操作人、原因、关联官方订单号、收款渠道备注并上传附件；顾问/教务端界面须屏蔽任何线下收款按钮。
- **需求修订建议**：
  1. 财务系统增加“线下拆分”专用功能，仅财务角色可见可用；拆分时强制填写拆分原因、关联官方订单号，并上传证明材料，记录操作日志。
  2. 拆分操作后，触发发票模块标记拆分状态，避免重复开票或金额错误。
  3. 顾问、教务等前端角色所有模块彻底移除“现金收款”“线下收款”入口；对账模块自动比对官方渠道流水与财务台账，线下拆分后推送财务手工核对一致性。
  4. 界定线下拆分适用场景白名单，并纳入审计范围。
- **新状态**：`待CCB裁决`（财务立场：同意划分边界，线下拆分仅限财务操作且全流程留痕，顾问端禁收线下款；需与课程顾问、教务侧确认界面屏蔽与合规闭环）

### 校长 补访澄清

- **ISSUE-008**
  - **一句话原疑点**：课程顾问要求特殊审批“1-2分钟内”批复，与校长对关键操作的强制严谨审批诉求冲突，可能导致销售效率与风控刚性失衡。
  - **涉众明确答复**：校长明确给出了量化分级方案：①单笔订单总额≤5000元且折扣≥9折，授权校区主管审批，超时30分钟自动通过但标记“异常通过”并抄送校长/财务；②总额>10000元、折扣<85折、超150课时或捆绑销售，强制校长审批，工作日4小时/非工作日8小时内处理，绝不允许自动通过；③校长指定教学总监和运营总监为一级代理人，5分钟未处理提醒，30分钟未处理自动转派值班校长，代理人15分钟内未处理则订单自动拒绝。
  - **需求修订建议**：
    1. 将原REQ-ORDER-003和REQ-APPROVAL-001合并修订，纳入校长给出的三级审批路由规则，包括金额、折扣、课时包类型阈值。
    2. 明确各审批节点的超时处理机制：主管节点（30分钟超时自动通过+异常标记）、校长节点（无超时通过+时限提醒）、代理人节点（15分钟超时自动拒绝+重审机制）。
    3. 增加“值班校长”和“代理人”配置功能。
  - **新状态**：`已澄清`

- **ISSUE-017**
  - **一句话原疑点**：看板“今日实时数据”与“延迟不超过5分钟”的要求存在模糊空间，不明哪些指标需强实时。
  - **涉众明确答复**：仍未明确
  - **需求修订建议**：建议在后续评审中由产品负责人结合本次校长对审批时效的敏感性推断，将“资金安全报警”、“审批超时状态”等指标定为1分钟内强实时，趋势类指标保持5分钟准实时。
  - **新状态**：`部分澄清`

- **ISSUE-018**
  - **一句话原疑点**：对账异常报警强制推送校长，但审批权限可下放至校区主管，导致责任与信息链条断裂。
  - **涉众明确答复**：未直接回应对账报警推送是否随之下放，但校长通过“异常通过”机制（即主管审批通过的超时订单会自动抄送校长和财务总监）间接强化了对下放权限的监控。对账异常本身仍是集团集中可见的报警，但校长有了下放审批事后抽查的抓手。
  - **需求修订建议**：
    1. 保留“资金对账报警集团集中可见”机制。
    2. 在REQ-CAMPUS-004中明确：由校区主管审批通过的资金变动订单，需单独在日终对账报告中标记，并在触发报警时一并推送该标记，使校长能直接追溯审批来源。
  - **新状态**：`部分澄清`

- **ISSUE-024**
  - **一句话原疑点**：校长定义的业务异常告警和系统管理员定义的高危安全告警可能重叠，打扰校长。
  - **涉众明确答复**：仍未明确
  - **需求修订建议**：基于校长本次对审批异常（如异常通过、超时拒绝）的高度关注，建议将审批流程相关异常（如主管30分钟超时通过、代理人15分钟超时拒绝）列为高优先级业务告警，与系统安全告警在同一告警中心按严重级别统一排序推送，避免普通业务趋势告警淹没关键审批风险。
  - **新状态**：`部分澄清`

### 系统管理员 补访澄清

- **ISSUE-019 + 原疑点**：退费规则变更时效矛盾——购买后规则变更，退费时按申请提交时间还是购买时规则未明确，可能导致计算偏差。  
- **涉众明确答复**：订单创建时必须自动保存购买当时的退费规则快照（含退费计算公式、阶梯比例表、手续费规则、赠课处理方式、退款上限、最短退款周期等），退费时强制使用该快照且不可实时查询系统配置表；规则变更后仅新订单生成新快照，历史订单快照永久不变；系统不提供任何修改历史快照的通用入口，极端监管需求只能通过“政策性强制退费”特殊流程处理（三级审批+影响报告）。  
- **需求修订建议**：在订单模型中增加 `order_refund_rules` 快照存储（JSON/字段化），退费计算接口必须读取该快照，不得读取当前配置表；定义快照必含的最小规则集：计算公式（剩余课时占比/已消耗倒扣）、阶梯退费比例、手续费（固定金额/百分比/封顶保底）、赠课处理（是否折现、折算单价）、退款上限、最短退款周期；退费审计日志需同时记录所使用的规则快照完整内容；增加“政策性强制退费”任务单流程，独立于常规退费，原订单标记关联标识。  
- **新状态**：已澄清  

- **ISSUE-021 + 原疑点**：系统管理员要求审计日志记录设备信息，财务审计需求未提及，可能导致缺少设备指纹、IP等安全追溯字段。  
- **涉众明确答复**：所有业务模块（含财务）的高风险操作必须记录统一的基础审计字段集，至少包括设备信息/设备指纹、IP、操作人、操作时间（精确到秒）、改前值、改后值；财务模块不能缩减基础字段，可在基础之上增加业务特有字段（如退款到账账户）；统一字段是硬性要求，需写入代码审查检查项。  
- **需求修订建议**：定义全局审计日志基础字段规范，设备指纹、IP、操作人ID、操作时间（秒级）、改前值、改后值为必填且非空；财务模块接口在记录退费、对账确认、人工冲销等高风险操作时强制携带上述字段，否则记录为异常日志；审计查询界面支持跨模块按设备指纹、IP联合筛选；在开发规范中纳入该字段检查项。  
- **新状态**：已澄清
