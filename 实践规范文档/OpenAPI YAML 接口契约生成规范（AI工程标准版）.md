# OpenAPI YAML 接口契约生成规范（AI工程标准版）

# 0 前置说明

本文为AI软件工程体系**接口层标准化契约规范**，承接上游TLCD三层约束、MDS模块边界、DTS依赖拓扑、ASD架构风格、RTM需求溯源基线，是系统对外/对内接口的唯一机器可读契约标准。

本规范基于 **OpenAPI 3\.0\.3 稳定版本** 统一语法、结构、语义、命名、错误码、传参、样例，杜绝自由编写导致的接口混乱、文档与代码不一致、联调返工、AI生成接口不统一等问题。

本规范与全体系工程产物同源闭环：**需求溯源→架构风格→模块边界→拓扑依赖→三层约束→OpenAPI契约→代码生成→接口校验**。

# 1 总则

## 1\.1 规范定位

OpenAPI契约（OAS）是系统接口能力的**唯一官方结构化定义**，用于统一前后端、服务间、模块间的调用协议、参数结构、数据类型、异常规则、返回格式。是AI自动生成Controller、接口代码、Mock数据、接口文档、自动化测试用例的唯一真值来源。

区别于零散接口文档、口头接口约定、代码注释文档，本规范要求：**契约先行、契约唯一、契约驱动代码、契约可机器校验**。

## 1\.2 核心价值

- **统一接口标准**：全局统一请求方式、路径、参数、结构体、错误码、响应格式，消除接口风格混乱。

- **杜绝接口漂移**：接口能力、字段结构、请求方式变更必须同步更新契约，防止代码与文档不一致。

- **AI可全自动驱动**：标准化YAML结构可直接驱动AI生成合规Controller、入参DTO、出参VO、校验逻辑、接口注释。

- **可自动化校验**：支持契约语法校验、参数合规校验、接口权限校验、拓扑依赖校验。

- **全链路溯源**：接口→模块→需求→架构约束双向溯源，确保每一个接口均有需求依据、架构依据。

- **降低联调成本**：前后端、服务间基于同一契约开发，无需反复对齐字段与格式。

## 1\.3 适用范围

适用于所有HTTP/HTTPS风格接口、内部模块调用接口、服务间BFF接口、对外开放接口的契约定义。

AI工程强制流程：**TLCD三层约束定稿 → OpenAPI契约编写评审 → 接口代码生成 → 接口校验**。无合规OpenAPI契约，禁止AI批量生成接口代码。

## 1\.4 前置联动与输出产物

- **前置依赖**：RTM需求基线、ASD架构风格、MDS模块划分、DTS依赖拓扑、TLCD三层约束。

- **平行联动**：接口清单、接口权限矩阵、字段字典。

- **下游输出**：AI生成接口代码、DTO/VO结构体、Mock接口、自动化测试用例、接口文档站点。

## 1\.5 基础管控规范（体系同源）

### 1\.5\.1 编号体系

契约文档编号：`OAS-项目简码-版本号`

单接口唯一ID：`API-四位流水号`，终身不变、新增递增、废弃保留、永不复用。

### 1\.5\.2 文档状态

草稿 / 正式生效 / 迭代过期 / 废弃，与ASD/MDS/DTS/TLCD版本状态强一致。

### 1\.5\.3 核心强制原则

- **契约先行原则**：先有契约，后有代码，禁止代码先行、文档后置补录。

- **溯源唯一原则**：每个接口必须归属唯一模块、对应唯一RTM需求、遵循唯一三层约束。

- **结构标准化原则**：所有YAML文件结构、字段、注释、样例、标签严格统一，禁止自定义格式。

- **可校验原则**：所有字段必填/非必填、类型、范围、正则、默认值必须可机器校验。

# 2 基础技术规范（强制）

## 2\.1 基准版本

全局固定使用：**OpenAPI 3\.0\.3**，禁止混用2\.0与3\.1\+版本，保证AI解析、工具解析一致性。

## 2\.2 文件规范

- 文件后缀统一为 `.yaml`，禁止\.yml混用；

- 编码统一为 UTF\-8；

- 缩进统一为 **2个空格**，禁止Tab；

- 键名统一小写、下划线分隔，禁止驼峰、大驼峰、拼音。

## 2\.3 接口命名规范（全局统一）

路径规则：`/api/{模块标识}/{版本}/{资源}/{操作}`

- 模块标识：取自MDS模块短标识，全局统一；

- 版本：固定 `v1/v2`，小写；

- 资源：名词复数，如 user、order、device；

- 操作：list、detail、create、update、delete、batch、export、import。

示例：`/api/user/v1/user/list`

## 2\.4 请求方法强制规范

- **GET**：查询、列表、详情、导出（无数据变更）；

- **POST**：新增、复杂查询、批量操作、导入；

- **PUT**：全量更新；

- **PATCH**：局部字段更新；

- **DELETE**：删除、逻辑删除。

禁止滥用POST做查询、禁止GET携带复杂Body参数。

## 2\.5 全局统一返回结构（强制）

所有接口必须统一外层响应结构，禁止自定义返回体：

- `code`：业务码（字符串/数字统一规范）；

- `message`：可读提示信息；

- `data`：业务主体数据（成功数据统一放此处）；

- `timestamp`：时间戳。

# 3 OpenAPI YAML 标准结构（必填10大固定模块）

所有正式契约文件必须完整包含以下结构，无缺省、无遗漏、无自定义删减。

## 3\.1 openapi 版本声明（首行固定）

固定写法：`openapi: 3.0.3`

## 3\.2 info 全局元数据（基线溯源）

必须包含：标题、描述、版本、作者、创建时间、关联OAS编号、关联模块ID、关联RTM条目。

## 3\.3 servers 服务地址

区分本地、测试、预发、生产多环境URL，统一配置。

## 3\.4 tags 模块标签（与MDS强对齐）

每个Tag对应MDS真实业务模块，Tag名称、释义必须与模块定位完全一致，用于接口模块归类与AI收敛。

## 3\.5 paths 接口路径定义（核心）

单接口完整字段包含：接口摘要、详细描述、标签、请求参数、请求体、响应状态码、响应体、异常说明、样例数据。

## 3\.6 components 公共组件（强制复用）

所有公共DTO、VO、枚举、分页结构体、统一返回体必须抽至components/schemas，禁止接口内重复定义。

## 3\.7 parameters 全局公共参数

token、traceId、timestamp、page、size 等全局参数统一全局声明。

## 3\.8 responses 全局统一响应

200、400、401、403、404、500 统一全局响应模板。

## 3\.9 security 权限校验规则

全局认证方式、接口独立免登/需登规则统一声明，对齐系统安全约束。

## 3\.10 x\-extension 工程扩展字段（AI工程专属）

用于机器解析、溯源、约束绑定，包含：接口API编号、归属模块MOD\-ID、关联RTM\-ID、约束溯源TLCD\-ID、接口访问等级、是否允许AI生成、是否对外暴露。

# 4 字段级强制编写规范

## 4\.1 参数位置规范

- **path**：资源唯一标识、主键ID；

- **query**：普通查询、分页、筛选条件；

- **header**：认证、链路追踪、全局头信息；

- **cookie**：登录态Cookie；

- **body**：新增、更新、复杂提交结构体。

## 4\.2 字段属性必填规范（每条字段必须齐全）

所有结构体字段必须包含：

- **type**：数据类型（string/number/integer/boolean/object/array）；

- **description**：完整中文释义，禁止空、禁止简写；

- **required**：true/false 精准判定；

- **example**：合法可运行样例值；

- **default**：存在默认值必须声明；

- **format**：日期、手机号、邮箱、密码等格式约束；

- **enum**：枚举值必须全部列出并说明含义；

- **min/max**：长度、数值范围约束。

## 4\.3 数据类型强制规范

- ID、编号、手机号、验证码统一使用**string**，防止前端精度丢失；

- 整数使用 integer、小数使用 number；

- 布尔统一 boolean，禁止0/1替代布尔语义；

- 时间统一 string \+ format: date / date\-time。

## 4\.4 分页结构强制统一

所有列表接口分页参数、分页返回结构体全局统一模板，禁止各自定义分页字段。

## 4\.5 错误码规范

业务错误码分层统一：系统级码 \+ 模块级码 \+ 业务场景码，与全局异常规范、TLCD约束对齐。

# 5 接口分层约束联动规则

## 5\.1 架构层约束对齐

接口通信范式（同步/异步、事件/直调）必须严格遵循ASD架构风格，禁止契约定义与顶层架构冲突。

## 5\.2 模块层约束对齐

接口归属模块唯一，接口能力不得超出MDS模块职责范围，禁止跨模块职责泄露、越界接口。

接口调用关系必须符合DTS依赖拓扑白名单，禁止契约定义非法跨域依赖。

## 5\.3 代码层约束对齐

契约结构直接对应工程DTO/VO分层，字段命名、类型、约束完全对齐代码层编码规范。

# 6 迭代与版本规范

- **小迭代**：字段说明、样例、非必填字段增补，小版本升级；

- **结构变更**：增删核心字段、修改字段类型、修改请求方式、修改路径，大版本升级；

- **接口废弃**：标记deprecated: true，保留文档、禁止新代码使用；

- **版本强同步**：OAS版本与基线、MDS、DTS、TLCD版本一一对应。

# 7 命名与归档规范

## 7\.1 文件命名

主契约文件：`OAS_Vx_系统名称_接口契约.yaml`

模块拆分契约：`OAS_Vx_模块名_接口契约.yaml`

## 7\.2 归档路径

`wiki/baselines/BL-YYYYMMDD-NN/oas/`

# 8 编写禁忌（红线）

- 禁止无MDS模块归属、无RTM需求依据新增接口；

- 禁止字段缺type、缺description、缺example、缺required；

- 禁止混用OpenAPI2\.0与3\.x语法；

- 禁止接口能力超出模块职责边界、违反DTS拓扑依赖；

- 禁止外层返回结构不统一、错误码不规范；

- 禁止契约与代码不一致、文档后置补录。

# 9 最简标准模板（可直接落地）

```yaml
openapi: 3.0.3
info:
  title: 系统接口契约
  description: 基于AI工程规范标准化OpenAPI契约
  version: V1
  x-oas-id: OAS-XXX-V1
servers:
  - url: http://localhost:8080/api
tags:
  - name: user
    description: 用户模块
paths:
  /api/user/v1/user/list:
    get:
      tags: [user]
      summary: 用户列表查询
      description: 分页查询系统用户基础信息，仅做查询，无数据变更
      operationId: userList
      x-api-id: API-0001
      x-mod-id: MOD-001
      x-rtm-id: RTM-XXXX
      parameters:
        - name: page
          in: query
          required: true
          type: integer
          description: 页码
          example: 1
        - name: size
          in: query
          required: true
          type: integer
          description: 每页条数
          example: 10
      responses:
        "200":
          description: 成功返回
          $ref: "#/components/responses/SuccessResp"
components:
  schemas:
    PageQuery:
      type: object
      description: 通用分页查询参数
      properties:
        page:
          type: integer
          description: 页码
          example: 1
        size:
          type: integer
          description: 每页条数
          example: 10
    PageResult:
      type: object
      description: 通用分页返回结构体
  responses:
    SuccessResp:
      description: 统一成功响应

```
