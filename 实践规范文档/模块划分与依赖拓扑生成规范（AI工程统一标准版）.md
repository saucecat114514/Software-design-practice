# 模块划分与依赖拓扑生成规范（AI工程统一标准版）

# 0 前置说明

本文档为**双规范合一文档**，包含两大核心工程产物标准：

1）MDS 模块划分方案生成规范（原有完整版，已对齐ADR/ASD体系）；

2）DTS 模块依赖拓扑生成规范（新增配套规范，同源同范式、同管控、同基线）。

二者为**强绑定成对产物**：MDS定义「模块有哪些、边界是什么」，DTS定义「模块之间怎么连、什么能连、什么禁止连」，共同构成中层架构可机器校验的完整基准，是CodeGraph正向图谱、逆向漂移检测的唯一数据源。

# 1 模块划分方案（MDS）生成规范

## 1\.1 规范定位

模块划分方案（Module Division Scheme，MDS）是AI软件工程全生命周期的**中层架构核心落地文件**。承接上游ASD架构风格声明、ADR架构决策、SRS需求基线，向下约束工程目录、代码结构、接口设计、数据库设计、AI代码生成、CodeGraph依赖校验。

区别于传统随意式模块拆分，本规范定义**可量化、可校验、可机器识别、防边界漂移、防职责泄露**的标准化拆分体系，彻底解决AI生成代码常见的模块混杂、职责越界、循环依赖、边界模糊、逻辑散落等工程顽疾。

本规范与RTM需求溯源规范、ADR架构决策规范、ASD架构风格声明规范、CodeGraph校验规范完全同源对齐，构成「需求—决策—风格—划分—拓扑—设计—代码—校验」完整闭环。

## 1\.2 核心价值

- **固化系统业务边界**：将零散需求收敛为高内聚、低耦合的标准化业务模块，从顶层杜绝职责混杂。

- **统一AI拆分逻辑**：为AI自动拆分模块、生成工程结构、落地业务代码提供唯一标准依据，避免随机拆分、风格混乱。

- **根治循环依赖与跨域穿透**：通过明确模块依赖拓扑、禁止依赖规则，从源头规避架构缺陷。

- **支撑全链路溯源**：实现RTM需求条目→业务模块→代码包→接口→数据结构的双向追溯。

- **支撑架构迭代与资产复用**：标准化模块可沉淀为业务资产，支持同品类系统快速复用、迭代升级。

- **支撑自动化架构校验**：为CodeGraph正向图谱构建、逆向漂移检测提供基准模块边界。

## 1\.3 适用范围

本规范适用于所有软件项目的初始化模块拆分、迭代模块调整、业务域重构、架构升级场景。

在AI工程工作流中，**必须在ASD生效之后、详细设计与代码生成之前完成MDS\+DTS同步评审归档**。无模块划分、无拓扑约束，禁止开展接口设计、数据库设计、批量AI代码生成。

适配分层单体、事件驱动、微服务、混合架构等所有主流架构风格。

## 1\.4 前置与联动产物（强依赖）

- **前置必选产物**：已生效ASD架构风格声明、已接受ADR架构决策、基线版SRS需求文档、RTM需求溯源矩阵。

- **同频联动产物**：DTS模块依赖拓扑、工程目录结构模板、模块职责约束清单。

- **下游输出产物**：接口契约文档、数据字典、代码工程结构、CodeGraph模块与依赖基准规则。

## 1\.5 基础管控规范（全局强制）

### 1\.5\.1 唯一编号体系

文档编号：`MDS-项目简码-版本号`，全局唯一、基线绑定、迭代升级、不复用、不删除。

模块子项唯一终身ID：`MOD-三位流水号`，新增递增、废弃保留、迭代不变、永不复用。

### 1\.5\.2 状态枚举

文档状态：草稿 / 正式生效 / 迭代过期 / 废弃。

单模块状态：正常启用 / 迭代调整 / 废弃下线 / 新增待落地。

### 1\.5\.3 权属权限

编写：架构师/AI基线智能体；评审：CCB\+技术负责人\+产品负责人；仅架构师与基线智能体可编辑迭代，生效需CCB封存。

### 1\.5\.4 四大不可突破拆分原则

- 高内聚低耦合原则：单一模块职责高度聚焦，跨模块依赖最小化；

- 需求全覆盖原则：所有SRS、RTM条目100%归属，无遗漏、无重叠；

- 边界唯一原则：任意业务能力、数据实体、流程逻辑仅归属一个主模块；

- 可校验可落地原则：所有边界、职责、依赖可转为AI规则与CodeGraph校验规则。

## 1\.6 AI适配版五维拆分方法论（唯一标准）

拆分优先级从高到低，所有模块拆分必须遵循本顺序，禁止主观打乱：

1. **业务域维度（最高）**：按核心业务能力、场景、角色切割；

2. **数据实体维度**：按数据主体、归属、生命周期收拢；

3. **变更频率维度**：高频迭代业务独立拆分，隔离稳定底座；

4. **依赖强度维度**：强关联逻辑同模块，弱关联能力拆隔离；

5. **非功能约束维度**：日志、缓存、消息、定时、安全等通用能力独立成基础模块。

### 1\.6\.1 二级模块分类（强制统一）

所有模块仅分为两类，禁止混杂：**业务模块（承载业务需求）**、**通用基础模块（全局技术支撑）**。

### 1\.6\.2 拆分禁止铁律

- 禁止单模块承载无关多业务职责；

- 禁止业务核心逻辑散落至基础模块；

- 禁止数据实体跨模块多头归属；

- 禁止存在无人认领需求、接口、数据表；

- 禁止模块隐性依赖、循环依赖、双向强依赖。

## 1\.7 MDS标准10字段结构化规范（机器可解析）

正式基线版本必须完整包含以下10个章节，无缺省、无模糊描述。

1. **基础元数据**：编号、名称、适配架构、状态、基线、关联ASD/ADR/SRS/RTM、责任人；

2. **拆分依据与整体思路**：结合ASD风格、五维拆分取舍、粒度控制、臃肿/细碎规避策略；

3. **全局模块总览清单**：表格化列出全部模块ID、名称、类型、状态、核心定位；

4. **单模块详细定义**：唯一职责、覆盖场景、归属实体、能力清单、禁止职责、RTM溯源绑定；

5. **模块依赖拓扑规范**：允许依赖、禁止依赖、方向约束、解耦规则（与DTS联动）；

6. **工程目录映射规则**：模块ID→包路径、分层代码归属、收敛与禁止散落规则；

7. **模块通信与调用规范**：本地调用/API/事件/消息场景、传参规范、同步异步分工、事务异常规则；

8. **边界漂移校验标准**：职责越界、数据漂移、非法依赖、代码散落的量化判定；

9. **迭代与调整规则**：新增、拆分、合并、下线、重构的触发条件与同步流程；

10. **质量验收标准**：可落地、可评审、可自动化校验的合规条件。

## 1\.8 生成与迭代流程

新建流程：前置基线校验 → 全域需求与实体梳理 → 五维拆分 → 边界校准去重去环 → 结构化编写 → 机器规则转化 → CCB评审归档 → 全链路同步。

迭代流程：微调小版本升级；结构重构/风格变更大版本升级，旧版冻结归档，强制同步DTS、ASD、ADR、工程模板、AI约束。

文件命名格式：`MDS_Vx_系统名称_模块划分方案.md`

归档路径：`wiki/baselines/BL-YYYYMMDD-NN/module/`

# 2 模块依赖拓扑（DTS）生成规范（新增配套标准版）

## 2\.1 规范定位

模块依赖拓扑方案（Dependency Topology Scheme，DTS）是**MDS的唯一配套拓扑契约**，是中层架构「结构合法性」的机器可读基线。MDS解决「模块边界是什么」，DTS解决「模块之间允许怎么依赖、禁止怎么依赖」。

DTS是CodeGraph正向依赖图谱、逆向依赖比对、架构漂移检测的**唯一真值来源**，用于杜绝AI自由生成导致的层次穿透、循环依赖、非法跨域调用、拓扑混乱等问题。

## 2\.2 核心价值

- **拓扑结构化、无歧义**：将模糊的「模块之间可以调用」转化为有向、可枚举、可校验的依赖边集合；

- **防AI架构乱生成**：AI每次生成代码必须遵循DTS允许依赖白名单，禁止生成黑名单依赖；

- **自动化漂移检测**：支持机器自动比对「设计拓扑」与「代码真实拓扑」，输出依赖违规、层级穿透、循环依赖问题；

- **支撑并行开发**：明确模块依赖顺序与调用契约，多人多AI子技能并行开发不互相越界；

- **支撑架构演化**：拓扑变更可版本追溯、可复盘、可沉淀为架构资产。

## 2\.3 前置联动与输出关系

- **强前置**：已生效ASD、已生效MDS、已接受架构ADR；

- **平行联动**：与MDS成对评审、成对归档、成对迭代；

- **下游输出**：CodeGraph正向依赖图谱、AI依赖约束提示词、流水线校验规则、接口依赖规范。

## 2\.4 DTS基础管控规范（与MDS/ADR/ASD同源）

### 2\.4\.1 编号体系

拓扑文档编号：`DTS-项目简码-版本号`，与对应MDS版本一一对应、同升同降。

依赖边唯一标识：`EDGE-四位流水号`，每条允许依赖、禁止依赖均有唯一ID，用于问题溯源。

### 2\.4\.2 文档状态

草稿 / 正式生效 / 迭代过期 / 废弃，与MDS状态强一致。

### 2\.4\.3 拓扑四大强制原则

- **有向可追溯原则**：所有依赖必须是有向边，明确「谁依赖谁」，禁止无向模糊描述；

- **白名单优先原则**：默认禁止一切未声明依赖，仅白名单依赖合法；

- **无环原则**：业务模块之间禁止出现直接/间接循环依赖；

- **层级合规原则**：严格遵循ASD架构层级单向依赖规则，禁止反向穿透、跨层穿透。

## 2\.5 依赖分类标准（统一机器识别维度）

所有模块依赖分为三类，必须逐条归类，用于差异化校验：

1. **强制允许依赖（白名单）**：业务模块→基础模块、合规跨业务调用，明确写入拓扑白名单，AI可正常生成；

2. **条件允许依赖**：仅可通过事件/抽象接口/网关中转，禁止直接硬依赖；

3. **绝对禁止依赖（黑名单）**：循环依赖、反向依赖、跨层穿透、越域私有调用，机器强制拦截。

## 2\.6 四大架构风格拓扑通用约束（强制式）

### 2\.6\.1 分层架构拓扑约束（默认主流）

- 层级单向依赖：Controller → Service → Repository，禁止反向；

- 业务模块可依赖公共基础模块，基础模块禁止反向依赖业务模块；

- 业务模块间尽量低耦合，如需联动优先事件/抽象接口，禁止直接Bean循环依赖。

### 2\.6\.2 事件驱动架构拓扑约束

- 生产者只发事件、不直接调用消费者；

- 消费者订阅事件反向解耦，禁止消费者强依赖生产者业务接口；

- 同步关键链路保留少量直调，异步链路全部事件化解耦。

### 2\.6\.3 微服务架构拓扑约束

- 服务间仅允许网关/API/消息依赖，禁止直连数据库、禁止内部包依赖；

- 服务自治、无双向循环依赖；

- 公共中台服务为单向被依赖方。

### 2\.6\.4 单体架构拓扑约束

- 内部模块严格边界隔离，禁止跨模块随意调用私有方法；

- 公共能力统一收敛基础模块，业务模块互斥低耦合。

## 2\.7 DTS标准结构化字段（机器可解析10字段）

正式生效DTS必须包含以下完整结构，适配图谱构建与自动化校验。

### 2\.7\.1 基础元数据

拓扑编号、对应MDS版本、对应ASD版本、基线编号、状态、更新日期、评审人。

### 2\.7\.2 拓扑整体策略

整体依赖规则、层级方向总约束、解耦策略、防环策略、AI生成总原则。

### 2\.7\.3 模块层级依赖矩阵（核心）

以矩阵表格明确：每层允许依赖、每层禁止依赖、穿透判定标准。

### 2\.7\.4 逐条有向依赖白名单（EDGE清单）

每条包含：EDGE编号、依赖起点模块、依赖终点模块、依赖方式（本地调用/API/事件/消息）、适用场景、约束说明。

### 2\.7\.5 条件依赖与解耦方案

所有不能直接硬依赖的场景，明确事件主题、抽象接口、网关路由、适配层方案。

### 2\.7\.6 绝对禁止依赖黑名单

明确禁止的循环组合、反向层级、跨域穿透、私有访问、非法硬依赖清单。

### 2\.7\.7 拓扑通信范式规范

统一同步/异步、调用/订阅、传参、事务、跨模块异常、缓存一致性拓扑规则。

### 2\.7\.8 拓扑漂移校验规则（对接CodeGraph）

包含：非法依赖漂移、层级穿透漂移、循环依赖漂移、拓扑缺失漂移、调用范式不匹配漂移的量化判定。

### 2\.7\.9 拓扑迭代与变更规则

依赖新增、删除、调整、解耦升级、事件改造的变更流程与基线同步要求。

### 2\.7\.10 拓扑验收标准

无环、无非法穿透、白名单全覆盖、黑名单无逃逸、可机器校验、与MDS边界一致。

## 2\.8 DTS标准生成流程

1. 前置校验：MDS/ASD/ADR基线已生效；

2. 梳理全部模块角色与层级归属；

3. 枚举所有必要业务依赖，形成有向白名单；

4. 筛查去环、去反向穿透、去跨域越界；

5. 补充条件解耦方案与禁止黑名单；

6. 生成CodeGraph校验规则与AI约束；

7. 与MDS联合评审、成对归档；

8. 同步更新AI提示词与自动化校验流水线。

# 3 MDS与DTS联动闭环规范（强制）

- **版本强一致**：MDS与DTS版本号同步迭代、同生效、同过期；

- **边界强对齐**：DTS依赖起点终点必须完全匹配MDS模块ID与模块职责；

- **问题双向追溯**：CodeGraph检出拓扑漂移，反向修正MDS边界或DTS规则；

- **AI统一消费**：AI生成代码同时读取MDS边界约束\+DTS拓扑约束，缺一不可。

# 5 编写禁忌（统一红线）

- 禁止无MDS先写DTS、无边界先定依赖；

- 禁止无向依赖、模糊依赖、口头依赖；

- 禁止存在未定义的隐性依赖、循环依赖；

- 禁止拓扑规则与ASD架构风格冲突；

- 禁止白名单不全、黑名单不严，导致AI自由生成架构。
