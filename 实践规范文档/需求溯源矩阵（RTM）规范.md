# 需求溯源矩阵（RTM）完整生成规范与工程产物细则（全生命周期标准版）

# 1 概述

## 1\.1 目的

需求溯源矩阵（RTM，Requirements Traceability Matrix）是软件需求工程**唯一双向追溯载体**，用于建立：业务需求、原始涉众需求、SRS需求、建模产物、设计、开发模块、测试用例、验收结果、变更记录、基线版本的全链路绑定关系。

本规范用于统一 AI 智能体自动生成、人工审核、CCB 基线管控、需求变更比对的标准化输出，彻底解决：需求遗漏、需求蔓延、变更影响不可评估、基线不可追溯、测试无覆盖依据等问题。

## 1\.2 适用范围

适用于：

- 初始需求基线建立

- 需求迭代更新

- CR 变更需求分析与基线比对

- CCB 变更评审

- 版本发布、验收、交付归档

- 全生命周期需求质量审计

## 1\.3 追溯标准（国标\+IEEE830 双标准）

遵循 GB/T 9385、GB/T 8567、IEEE 830 双向追溯要求：

- **正向追溯**：业务目标 → 原始需求 → SRS → 设计 → 代码 → 测试 → 验收

- **反向追溯**：缺陷/测试项/代码模块 → 对应SRS需求 → 原始业务诉求

# 2 核心定义与术语规范

## 2\.1 四级需求编号体系（全局唯一、终身不变）

所有需求 ID **一旦生成，永久不修改、不复用、不删除**，废弃仅标记状态。

- **BR**：Business Requirement 业务目标需求（顶层）

- **UR**：User Requirement 涉众原始需求（AI收集/CR提交）

- **FR**：Functional Requirement 软件功能需求（SRS正式条目）

- **NFR**：Non\-Functional Requirement 非功能需求

- **IFR**：Interface Requirement 接口需求

- **CR**：Change Request 变更需求单号

编号格式统一：`类型-模块-三位流水号`

示例：BR\-SYS\-001、UR\-USER\-002、FR\-BIZ\-003、NFR\-PERF\-001

## 2\.2 需求状态枚举（强制标准，AI自动识别）

所有基线比对、RTM 更新只能使用以下 4 种状态：

1. **新增**：当前基线首次出现的需求

2. **无变更**：与上一基线内容、规则、验收标准完全一致

3. **修改**：字段、逻辑、规则、阈值、流程、约束任意一项变更

4. **删除**：本版本基线作废、下线、不再实现

## 2\.3 变更来源枚举（固定）

- 涉众AI智能体沟通获取

- CR正式变更文档提交

- 内部质量优化

- 合规/政策强制变更

4. **SRS需求正式描述**：SRS原文完整摘录

5. **验收标准**：可量化、可测试、无歧义

6. **需求优先级**：P0必现/P1重要/P2次要

### 【13\-16：基线比对差异层（变更核心）】

1. **本次基线需求状态**：新增/无变更/修改/删除

2. **变更来源**：固定枚举

3. **变更差异详情**：精准对比新旧文本差异

4. **变更影响范围**：模块/接口/数据/权限/流程

1. **关联建模产物ID**：PlantUML用例/活动/时序图编号

2. **关联设计文档ID**：概要设计/详细设计章节号

3. **关联开发模块**：后端服务/前端页面/数据表

4. **数据字典关联ID**：关联数据字段编号

### 【21\-22：测试验收追溯层】

1. **关联测试用例ID**：TC\-xxx 全覆盖用例编号

2. **验收测试状态**：未测试/测试通过/测试失败/阻塞

# 4 RTM 生成流程规范（AI全自动执行步骤）

完全适配你现有的：需求获取→分析→建模→SRS→基线比对→新基线流程

## 4\.1 初始基线 RTM 生成流程（首次建基线）

1. A1 收集所有涉众原始需求，生成 UR 编号

2. A2 质量分析过滤无效需求

3. A3/A4 建模 \+ 生成正式SRS，分配 FR/NFR/IFR 编号

4. A6 基线智能体：建立 BR→UR→SRS 三级绑定

5. 填充优先级、验收标准、建模关联ID

6. 状态统一标记为【新增】

7. CCB审核通过后归档为初始基线 RTM

## 4\.2 变更基线 RTM 生成流程（核心重点）

每次CR变更必须完整执行以下步骤，缺一不可：

1. **加载旧基线全集**：读取上一版本完整 RTM \+ 旧SRS

2. **新旧逐条比对**：字段级、规则级、文本级精准 diff

3. **自动标记4种状态**

4. **保留所有旧ID**：绝不删除历史条目

5. **新增需求分配新ID**：不占用旧作废ID

6. **生成差异文本**：逐条输出“变更前后对比”

7. **更新追溯链路**：更新建模、设计、模块关联

8. **生成新版RTM**

9. **联动生成《需求差异文档》**

10. **随新基线归档**

## 4\.3 强制约束（不可违反）

- 禁止需求ID复用

- 禁止覆盖历史RTM

- 禁止缺失追溯链路（上游必须可查）

- 禁止模糊描述，差异必须字段级精准

- 删除的需求必须保留行，仅标记状态，用于审计

# 10 RTM 权限与管控规范

## 10\.1 角色权限划分

为保障RTM基线数据安全性、唯一性、规范性，严格执行分级权限管控，不同角色仅可操作对应权限范围内的内容，禁止越权修改、删除基线数据。

- **研发工程师**：仅可查看RTM全量内容、补充自身负责模块的开发模块信息，无修改、新增、删除需求条目权限，可提交内容修正工单至CCB。

- **测试工程师**：可查看全量内容、回填测试用例ID与验收测试状态，禁止修改需求描述、验收标准、基线状态等核心字段。

- **需求工程师/AI基线智能体**：拥有RTM新增、编辑、比对、版本生成权限，负责日常更新、差异校对、链路完善，是RTM日常维护主体。

- **CCB评审组**：拥有最终审核、基线封存、版本生效权限，可驳回不规范RTM产物，审批通过新版基线生效。

- **项目负责人**：全量查看权限、变更审批权限，无直接编辑权限，监督基线整体质量。

## 10\.2 版本锁定机制

已封存的正式基线版本**永久锁定**，禁止任何形式的直接修改、覆盖、删除。若基线存在错误、遗漏问题，必须通过正式CR变更流程，生成新版基线迭代更新，历史错误版本永久归档留存，用于问题溯源与责任界定。

# 11 RTM 异常场景处理规范

针对项目全周期常见的需求异常场景，制定统一标准化处理规则，杜绝随意处置、链路断裂、数据混乱等问题。

## 11\.1 无效/废弃需求处理

经需求评审、CCB判定为无效、冗余、无需实现的需求，禁止直接删除RTM条目，统一标记状态为**删除**，在差异详情栏标注废弃原因、评审时间、审批编号，保留完整溯源记录，用于后续审计复盘。

## 11\.2 紧急变更需求处理

线上故障修复、合规紧急整改等紧急变更需求，可启动加急评审流程，但**不得豁免RTM更新**。需在变更落地后24小时内，完成新旧基线比对、RTM更新、差异文档生成、产物归档，补齐全流程溯源链路。

## 11\.3 需求拆分/合并处理

- **需求拆分**：原单条SRS需求拆分为多条子需求时，原需求标记为【修改】，新增子需求分配全新独立ID，在差异详情中注明拆分关系，新旧链路双向关联。

- **需求合并**：多条旧需求合并为一条新需求时，所有旧需求标记为【删除】，新增合并后需求分配新ID，明确标注合并来源，保留所有历史溯源信息。

## 11\.4 漏项需求补录处理

基线封存后发现需求漏项，禁止直接插入旧基线，需在当前最新迭代版本中新增需求条目，标记为【新增】，在变更来源与差异详情中明确标注为历史漏项补录，同步补齐建模、设计、测试全链路关联信息。

# 12 周期核查与复盘规范

## 12\.1 日常核查（迭代周期内）

每个迭代周期结束前，需求工程师联合测试负责人完成RTM自查，重点核查：需求链路完整性、状态准确性、字段完整性、测试用例覆盖率，整改所有链路缺失、状态错误、字段模糊问题，确保迭代基线合规。

## 12\.2 版本复盘（正式发布前）

系统版本正式发布前，CCB组织全量RTM基线复盘，核对全套工程产物完整性、变更影响范围准确性、需求验收闭环情况，无问题后方可审批发布。

## 12\.3 季度审计（长期运维）

每季度开展一次基线全量审计，梳理所有历史基线版本，核查ID规范性、变更记录完整性、废弃需求归档情况，输出季度基线质量审计报告，优化需求管控流程。

# 13 常见问题FAQ（落地答疑）

- **Q1： minor微调（文案优化、格式修正）是否需要更新RTM？**
**A1：需要**。所有SRS原文、描述、规则的字段级修改，无论改动大小，均需标记【修改】状态，记录差异详情，纳入基线变更，禁止私改不归档。

- **Q2：测试用例新增、优化是否需要联动更新RTM？**
**A2：需要**。测试用例新增、修改、作废后，需同步更新RTM对应测试用例ID、验收状态，保证需求与测试链路实时同步。

- **Q3：不同模块的相似需求是否可以复用ID？**
**A3：禁止**。所有需求ID全局唯一，跨模块、跨版本、跨迭代均不可复用，废弃ID永久封存，仅可标记状态。

- **Q4：初始基线无历史对比版本如何填写差异字段？**
**A4：统一填写“初始基线，无历史版本”，需求状态全部标记为【新增】，无需生成差异比对文档。**

- **Q5：非功能性需求、接口需求是否需要纳入RTM？**
**A5：必须纳入**。NFR、IFR与FR需求同等管控，100%入矩阵，完整覆盖溯源链路，不允许遗漏非功能、接口类需求。

# 14 规范生效与迭代说明

本《需求溯源矩阵（RTM）完整生成规范与工程产物细则（全生命周期标准版）》自发布之日起正式生效，覆盖项目所有存量、增量需求基线。

本规范支持版本迭代优化，后续流程、字段、管控规则调整，需通过正式评审后更新规范文档，历史基线不受新规范影响，仅新基线执行最新规范。

所有项目参与人员必须严格遵照本规范执行RTM生成、更新、评审、归档工作，违规操作纳入项目质量考核。
